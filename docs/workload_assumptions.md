# AIRMAN Skynet Pilot Workload Assumptions

## 1. Purpose of This Document

This document explains the expected workload for the Skynet Ops Audit Service
used in the Cloud Operations Intern assessment.

Workload assumptions help determine:

- how much traffic the service receives
- how infrastructure should scale
- what monitoring thresholds should be used
- how much cloud cost to expect

These assumptions are based on a **small pilot deployment** rather than a
large production system. The goal is to design a system that is reliable,
cost-efficient, and operationally simple.

---

## 2. Deployment Context

The Skynet Ops Audit Service is assumed to be deployed as part of a **pilot rollout**
of the AIRMAN Skynet platform.

The pilot deployment is expected to support:

- 1 to 3 flight training academies
- limited internal usage by administrators and staff
- early testing and operational monitoring

Because the system is running on **startup cloud credits**, the design should avoid
unnecessary spending and focus on efficient resource usage.

The infrastructure should support moderate usage while remaining easy to maintain
and inexpensive to run.

---

## 3. Functional Role of the Service

The Skynet Ops Audit Service is responsible for collecting and retrieving
operational audit events.

These events represent system activity that administrators may want to review.

The service supports the following API endpoints:

POST /events  
Used to submit a new audit event.

GET /events  
Used to retrieve previously stored events.

GET /health  
Used by monitoring systems and load balancers to verify that the service is running.

This service does not represent the full Skynet platform but serves as a
representative **operations support component**.

---

## 4. Traffic Assumptions

The expected daily request volume for the service is between:

5,000 and 20,000 requests per day.

This includes:

- event ingestion requests
- event retrieval requests
- health checks
- monitoring probes
- occasional testing traffic

Traffic is **not evenly distributed during the day**.

Typical traffic patterns:

- higher activity during academy operating hours
- lower activity overnight
- reduced activity during weekends

Approximately **70% of the total daily traffic occurs within a 10-hour window**,
usually during active working hours.

This means the system should handle occasional traffic spikes
but does not require extreme scaling capabilities.

---

## 5. Request Distribution

The different API endpoints are expected to receive traffic in the following ratio.

POST /events  
Approximately 55% of all requests.

This endpoint is used to ingest audit events from the system.

GET /events  
Approximately 35% of requests.

This endpoint retrieves stored events for administrators.

GET /health  
Approximately 10% of requests.

These requests are generated by monitoring systems and the load balancer
to verify service health.

Understanding this distribution helps determine how the system handles
write operations versus read operations.

---

## 6. Payload Size Assumptions

The size of requests and responses affects network usage,
processing time, and storage requirements.

POST /events requests contain JSON payloads describing system events.

Typical payload size:

0.5 KB to 3 KB

Occasionally larger payloads may occur when additional metadata is included,
which can increase payload size up to approximately 5 KB.

GET /events responses return multiple stored records.

The size of the response depends on the query limit parameter.

Typical response size:

5 KB to 40 KB

This assumes a default limit of around 20 events per request.

These assumptions help estimate network usage and storage growth.

---

## 7. Data Growth Expectations

The system stores audit events generated by users and system components.

Expected ingestion rate:

200 to 2,000 events per day.

Because each event is relatively small, the overall storage growth remains modest.

Estimated monthly storage growth:

10 MB to 250 MB per month (raw event data).

This estimate includes only the event records themselves.

Additional storage may be required for:

- logs
- monitoring metrics
- backups
- indexing overhead

These additional factors should be considered when estimating total storage cost.

---

## 8. Availability Expectations

Since this is a **pilot environment**, the system does not require
enterprise-level availability.

The acceptable target availability is approximately:

99% uptime.

This means occasional downtime is acceptable as long as
the system can recover quickly and the operational procedures
are clearly documented.

The emphasis for this assessment is on:

- clear operational procedures
- effective monitoring
- quick troubleshooting
- reasonable system reliability

rather than extremely complex high-availability architectures.

---

## 9. Latency Expectations

Latency refers to how quickly the system responds to requests.

The expected response times for this service are:

GET /health  
Typical response time: less than 200 milliseconds.

POST /events  
Typical response time: less than 500 milliseconds.

GET /events  
Typical response time: less than 1000 milliseconds.

The retrieval endpoint may take longer because it queries stored data.

Monitoring thresholds and alerts should be designed based on these expectations.

---

## 10. Environment Assumptions

At minimum, the system includes:

1 development environment.

The development environment is used for testing and debugging.

To reduce cloud costs, development resources may be:

- stopped outside working hours
- scaled down when not needed

A staging environment may be added later as an improvement
but is not required for the pilot deployment.

---

## 11. Logging and Monitoring

The system should produce structured application logs
to simplify debugging and operational analysis.

Logs are collected using AWS CloudWatch Logs.

Suggested log retention policies:

Development environment logs: 3 to 7 days

Pilot or staging environment logs: 7 to 14 days

Shorter retention periods help reduce storage costs
while still allowing enough time to diagnose issues.

Monitoring dashboards and alerts should track:

- request volume
- service latency
- error rates
- system health

---

## 12. Cost Target and Optimization

AIRMAN is operating on startup cloud credits, so controlling
cloud costs is an important part of the system design.

The expected monthly cost target for the pilot deployment is:

$25 to $75 per month.

To meet this target, the infrastructure should:

- avoid idle compute resources
- use small instance sizes where possible
- minimize unnecessary services
- control log and monitoring data retention
- avoid expensive networking configurations

Regular cost monitoring and cleanup procedures
should be implemented to prevent unexpected charges.
